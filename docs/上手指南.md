# health-store 上手指南

> 个人健康档案管理系统 — 上传检查单据、AI 解读、趋势追踪。

---

| 工具 | 版本 | 备注 |
|------|------|------|
| Node.js | ≥ 20 | `node --version` |
| pnpm | ≥ 9 | `pnpm --version` |
| Python | ≥ 3.10 | `python3 --version` |

## 一次性安装（首次）

### 1. Web 依赖

```bash
cd health-store
pnpm install
```

### 2. 建数据库

```bash
pnpm db:generate   # 生成 migration（已生成可跳过）
pnpm db:migrate    # 在 data/health.db 建表
```

### 3. 配置 LLM

编辑 `apps/web/.env.local`，把 DeepSeek API Key 填进去（从 `../self-wiki/.env` 里的 `DEEPSEEK_API_KEY` 复制）：

```env
OPENAI_PROVIDER_NAME=deepseek
OPENAI_BASE_URL=https://api.deepseek.com/v1
OPENAI_API_KEY=sk-...
OPENAI_MODEL=deepseek-v4-flash
OCR_SERVICE_URL=http://localhost:8700

# 可选：指定 SQLite 数据库路径；默认是 apps/web/../../data/health.db
DATABASE_PATH=../../data/health.db

# Pipeline 运行日志：默认记录摘要，不记录 OCR/LLM 全文
PIPELINE_LOG_ENABLED=true
PIPELINE_LOG_FULL_TEXT=false

# OCR → 抽取前的 LLM 纠错层：默认开启
LLM_REPAIR_ENABLED=true
```

想换其他 provider？改这三行就够：

```env
# Anthropic Claude
OPENAI_PROVIDER_NAME=anthropic
ANTHROPIC_API_KEY=sk-ant-...
OPENAI_MODEL=claude-sonnet-4-6
# （不需要 OPENAI_BASE_URL）

# 本地 Ollama
OPENAI_PROVIDER_NAME=ollama
OPENAI_BASE_URL=http://localhost:11434/v1
OPENAI_API_KEY=ollama
OPENAI_MODEL=qwen2.5:7b
```

### 4. OCR 服务依赖

```bash
cd services/ocr
uv venv --python 3.10    # 创建 venv
uv pip install -r requirements.txt  # 从全局缓存硬链接，已下载过的包近乎瞬间完成
```

> 没有 `uv`？`pip install uv` 或 `brew install uv` 安装，之后所有项目共享同一个包缓存（`~/.cache/uv`），和 pnpm 一样不重复下载和占用磁盘。

OCR 默认使用 PaddleOCR-VL，并在 Apple Silicon 上走 CPU 推理：

```bash
export OCR_ANALYSIS_MODE=vl          # 默认值；结构化 Markdown + 表格
export PADDLEOCR_DEVICE=cpu          # 默认值；PaddlePaddle 无 Metal/MPS 后端
```

需要回退到旧 PP-OCRv6 行为时：

```bash
export OCR_ANALYSIS_MODE=ppocr
```

如果使用远端 VL 识别后端，可选设置：

```bash
export PADDLEOCR_VL_REC_BACKEND=...
export PADDLEOCR_VL_REC_SERVER_URL=...
```

## 日常启动

**两个终端分开跑：**

```bash
# 终端 1：Web（含 Next.js 热更新）
pnpm dev:web

# 终端 2：OCR 微服务（端口 8700）
cd services/ocr
uv run uvicorn main:app --reload --port 8700
```

或者一条命令同时启动（需要 concurrently）：

```bash
pnpm dev
```

打开 http://localhost:3000

## 数据库结构速查

| 表 | 用途 |
|----|------|
| `documents` | 上传的检查单据（原图路径、OCR 结果） |
| `metric_catalog` | 标准指标词典（ALT/谷丙转氨酶/丙氨酸氨基转移酶 → 同一条目） |
| `measurements` | 归一化后的时序测量值（趋势分析核心） |
| `health_logs` | 饮食/运动/睡眠/症状等日常记录 |
| `wearable_samples` | Apple 健康导出数据 |
| `notes` | 非结构化笔记（AI 自动分类） |
| `tags` | 统一标签体系 |
| `tag_relations` | 标签与各类实体的关联 |
| `insights` | AI 生成的洞察报告 |

## OCR 服务接口

```bash
# 健康检查
curl http://localhost:8700/health

# 解析一张化验单图片
curl -X POST http://localhost:8700/parse \
  -F "run_id=manual-test-001" \
  -F "file=@/path/to/report.jpg"
# 返回：{ "markdown": "...", "analysis_text": "...", "json_data": [...], "mode": "vl", "run_id": "...", "timing": {...}, "blocks": [...] }
```

上传、重解析和批量导入会把 OCR 与 LLM 抽取写入 `pipeline_runs` 表，并追加摘要 JSONL 到 `data/logs/ocr-runs.jsonl` / `data/logs/llm-runs.jsonl`。如需临时关闭日志：

```env
PIPELINE_LOG_ENABLED=false
```

默认链路是 `OCR Markdown → LLM 纠错 → LLM 结构化抽取`。纠错失败会记录 `llm_repair` 错误日志，并回退原始 OCR 文本继续抽取；如需 A/B 对比纠错前后效果：

```env
LLM_REPAIR_ENABLED=false
```

## 常见问题

**OCR 首次运行很慢？**
正常，首次调用 `PPStructureV3` 时会下载 PP-OCRv6 模型（约 100–400MB，存在 `~/.paddleocr/`）。之后调用会复用缓存，速度快很多。

**换了新的 LLM provider 不生效？**
改 `.env.local` 后需要重启 `pnpm dev:web`，Next.js 不会热更新 env 文件。

**数据库在哪，怎么备份？**
`data/health.db` 和 `data/uploads/`，定期用 `cp -r data/ ~/backup/health-$(date +%Y%m%d)/` 备份即可。
